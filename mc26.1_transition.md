# Переход на Minecraft 26.1 — план и требования

Документ описывает всё необходимое для добавления поддержки MC 26.1 в проект ResourceTracker.
**Старые профили (legacy / transition / modern) продолжают жить и развиваться параллельно.**

Источники: официальная документация Fabric — [Porting to 26.1](https://docs.fabricmc.net/develop/porting/), [Migrating Mappings](https://docs.fabricmc.net/develop/porting/mappings/), [Fabric API 26.1 Renames](https://docs.fabricmc.net/1.21.11/26.1/develop/porting/fabric-api).

---

## Что изменилось в MC 26.1

### Versioning

Mojang перешли на схему `YY.drop` (год.релиз):

| Старая схема | Новая схема |
|---|---|
| 1.21.5, 1.21.11, … | 26.1, 26.2, 27.1, … |
| `26w01a` (снапшот) | `26.1-snapshot-1` |

1.21.11 — **последняя версия в старой схеме**.

### Деобфускация

MC 26.1 поставляется **полностью без обфускации**:

- JAR игры содержит читаемые имена классов/методов/полей прямо в байткоде
- Mojang больше не выпускает отдельные `.txt`-файлы маппингов
- **Yarn прекратил поддержку** — 1.21.11 последний выпуск
- Для 26.1+ строка `mappings` в `build.gradle` **удаляется полностью** — Loom подхватывает имена из JAR напрямую

### Системные требования

| Компонент | Было (1.21.x) | Стало (26.1) |
|---|---|---|
| Java | 21 | **25** |
| IntelliJ IDEA | любая | **2025.3+** (для Java 25) |
| Loom plugin ID | `fabric-loom` / `net.fabricmc.fabric-loom-remap` | **`net.fabricmc.fabric-loom`** |
| Mappings строка | `"net.fabricmc:yarn:…:v2"` | **убирается** |
| `modImplementation` | да | **`implementation`** |
| `modCompileOnly` | да | **`compileOnly`** |
| `remapJar` task | используется | **`jar`** (remapJar исчезает) |

## Два этапа перехода

### Этап 1 — Миграция маппингов (Yarn → Mojang)

**Инструмент:** `migrateMappings` (Loom Gradle task)

Переименовывает Yarn-имена в исходниках на Mojang-имена. Не поддерживает Kotlin-код. После запуска нужна ручная проверка — особенно Mixin-ов (у нас их нет).

**Что НЕ переименовывает migrateMappings:**
- GLFW-константы (`GLFW.GLFW_KEY_M`, …)
- JOML-методы (`pushMatrix()`, `translate()`, `scale()`)
- **Fabric API имена** — их нужно переименовывать вручную по отдельному списку (см. ниже)

### Этап 2 — Обновление Fabric API (ручные правки)

Fabric API 26.1 переименовал классы/методы под Mojang-имена. `migrateMappings` их **не трогает**. Нужно пройти по списку рисков ниже вручную.

---

## Конкретные рисковые правки для ResourceTracker

### Fabric API — рисковые рефакторинги (ручные, не migrateMappings)

| Старое имя (1.21.x) | Новое имя (26.1) | Вероятность | Файлы |
|---|---|---|---|
| `net.fabricmc.fabric.api.client.keybinding.v1.KeyBindingHelper` | `net.fabricmc.fabric.api.client.keymapping.v1.KeyMappingHelper` | **Высокая** | `nextgen/VersionCompat.java` |
| `KeyBindingHelper.registerKeyBinding(…)` | `KeyMappingHelper.registerKeyMapping(…)` | **Высокая** | `nextgen/VersionCompat.java` |
| `ClientTickEvents.START_WORLD_TICK` | `ClientTickEvents.START_LEVEL_TICK` | **Высокая** | `ResourceTrackerClient.java` |
| `ClientTickEvents.END_WORLD_TICK` | `ClientTickEvents.END_LEVEL_TICK` | **Высокая** | `ResourceTrackerClient.java` |
| `ClientTickEvents$StartWorldTick` | `ClientTickEvents$StartLevelTick` | **Высокая** | `ResourceTrackerClient.java` |
| `HudRenderCallback` | **без изменений** (не в списке рефакторингов) | — | `ResourceTrackerClient.java`, `HudOverlay.java` |

### Mojang-имена (обрабатывает migrateMappings)

| Yarn-имя (текущее) | Mojang-имя (26.1) |
|---|---|
| `KeyBinding` | `KeyMapping` |
| `client.world` | `client.level` |
| `client.player.age` | `client.player.tickCount` |
| `Registries.ITEM` | уточнить после migrateMappings |
| `DataComponentTypes.CONTAINER` | уточнить после migrateMappings |
| `DataComponentTypes.BUNDLE_CONTENTS` | уточнить после migrateMappings |
| `SoundEvents.UI_BUTTON_CLICK` | уточнить после migrateMappings |
| `client.options.hudHidden` | уточнить после migrateMappings |

**Не затрагиваются migrateMappings:**
- GLFW-константы (`GLFW.GLFW_KEY_M`, `GLFW_KEY_UNKNOWN`, …)
- JOML-методы (`pushMatrix()`, `popMatrix()`, `translate(x, y)`, `scale(sx, sy)`)

## Файлы и конкретные изменения

### `gradle/wrapper/gradle-wrapper.properties`

Обновить Gradle командой (не редактировать вручную):

```bash
JAVA_HOME="C:/Program Files/Eclipse Adoptium/jdk-25.0.2.10-hotspot" \
  ./gradlew wrapper --gradle-version latest -b build-nextgen.gradle
```

### `build-nextgen.gradle` (отдельный файл для nextgen)

Ключевые отличия от `build.gradle`:

```diff
 plugins {
-    id 'fabric-loom' version '...'          // старый ID (с remap)
+    id 'net.fabricmc.fabric-loom' version '...'  // новый ID (без remap, для 26.1)
 }

 dependencies {
     minecraft "com.mojang:minecraft:${minecraft_version}"
-    mappings "net.fabricmc:yarn:${yarn_mappings}:v2"   // УБРАТЬ строку полностью
     modImplementation "net.fabricmc:fabric-loader:${loader_version}"

-    modImplementation "net.fabricmc.fabric-api:fabric-api:${fabric_version}"
+    implementation "net.fabricmc.fabric-api:fabric-api:${fabric_version}"  // mod* → без mod*
 }

-def targetJavaVersion = 21
+def targetJavaVersion = 25

 java {
-    sourceCompatibility = JavaVersion.VERSION_21
-    targetCompatibility = JavaVersion.VERSION_21
+    sourceCompatibility = JavaVersion.VERSION_25
+    targetCompatibility = JavaVersion.VERSION_25
 }
```

Также в `build-nextgen.gradle`:
- Заменить все `modImplementation` на `implementation`
- Заменить все `modCompileOnly` на `compileOnly`
- Заменить все `modLocalRuntime` на `localRuntime`
- Убрать задачу `remapJar` — для 26.1 финальный JAR это просто `jar`

### `src/compat/nextgen/…/VersionCompat.java` (новый файл)

Стартовая точка — копия `modern/VersionCompat.java`. После `migrateMappings` имена MC-классов обновятся автоматически; Fabric API рефакторинги — вручную:

```java
package net.fabricmc.resourcetracker.compat;

import net.minecraft.client.gui.DrawContext;
import net.minecraft.client.KeyMapping;           // было KeyBinding (после migrateMappings)
import net.minecraft.client.util.InputUtil;
// ВАЖНО: пакет изменился с keybinding → keymapping
import net.fabricmc.fabric.api.client.keymapping.v1.KeyMappingHelper;
import org.lwjgl.glfw.GLFW;

/**
 * Compatibility layer for Minecraft 26.1+.
 * Mojang-native names (no Yarn). Java 25. No obfuscation in game JAR.
 */
public class VersionCompat {

    public static void push(DrawContext ctx) {
        ctx.getMatrices().pushMatrix();
    }

    public static void pop(DrawContext ctx) {
        ctx.getMatrices().popMatrix();
    }

    public static void translate(DrawContext ctx, float x, float y) {
        ctx.getMatrices().translate(x, y);
    }

    public static void scale(DrawContext ctx, float sx, float sy) {
        ctx.getMatrices().scale(sx, sy);
    }

    public static KeyMapping registerOpenKey() {
        return KeyMappingHelper.registerKeyMapping(new KeyMapping(
                "key.resourcetracker.open",
                InputUtil.Type.KEYSYM,
                GLFW.GLFW_KEY_M,
                KeyMapping.Category.MISC    // уточнить после migrateMappings
        ));
    }

    public static KeyMapping registerToggleHudKey() {
        return KeyMappingHelper.registerKeyMapping(new KeyMapping(
                "key.resourcetracker.toggle_hud",
                InputUtil.Type.KEYSYM,
                GLFW.GLFW_KEY_UNKNOWN,
                KeyMapping.Category.MISC
        ));
    }
}
```

## Порядок реализации

```
1. git checkout -b mc-26.1-migration  (уже сделано)

2. Найти актуальные версии на https://fabricmc.net/develop/:
   - Fabric Loader 26.1
   - Fabric API 26.1
   - ModMenu 26.1 (может не выйти сразу)
   - Loom последний стабильный

3. Создать build-nextgen.gradle:
   - Скопировать build.gradle
   - Поменять plugin ID: 'fabric-loom' → 'net.fabricmc.fabric-loom'
   - Убрать строку mappings
   - modImplementation → implementation, modCompileOnly → compileOnly
   - Java 21 → Java 25
   - Убрать remapJar-специфичный код

4. Обновить Gradle wrapper до последней версии:
   JAVA_HOME="C:/Program Files/Eclipse Adoptium/jdk-25.0.2.10-hotspot" \
     ./gradlew wrapper --gradle-version latest -b build-nextgen.gradle

5. Создать src/compat/nextgen/…/VersionCompat.java
   (копия modern/, с KeyMappingHelper вместо KeyBindingHelper)

6. Первый compile — скачает 26.1 JAR:
   JAVA_HOME="C:/Program Files/Eclipse Adoptium/jdk-25.0.2.10-hotspot" \
     ./gradlew compileJava -b build-nextgen.gradle \
       -Pminecraft_version=26.1 -Pfabric_version=<...>
   (упадёт с ошибками — это нормально)

7. Запустить migrateMappings (Yarn → Mojang в src/main/java + src/compat/nextgen):
   JAVA_HOME="C:/Program Files/Eclipse Adoptium/jdk-25.0.2.10-hotspot" \
     ./gradlew migrateMappings -b build-nextgen.gradle \
       -Pminecraft_version=26.1 -Pfabric_version=<...>

8. Вручную исправить Fabric API рефакторинги (migrateMappings их не трогает):
   - KeyBindingHelper → KeyMappingHelper (пакет + класс + метод)
   - START_WORLD_TICK → START_LEVEL_TICK
   - END_WORLD_TICK → END_LEVEL_TICK

9. Повторить compile, исправить оставшиеся ошибки компиляции

10. Убедиться что старые профили не сломаны:
    ./gradlew compileJava -Pprofile=modern -Pminecraft_version=1.21.11 \
      -Pyarn_mappings="1.21.11+build.4" -Pfabric_version="0.141.3+1.21.11"

11. build.yml → 4-й entry с nextgen, per-entry java_version

12. In-game тест (./gradlew runClient -b build-nextgen.gradle ...)
```

## Параллельная разработка старых версий

Поскольку профили полностью изолированы через `src/compat/{profile}/`, работа над legacy/transition/modern продолжается как прежде:

- Багфиксы и фичи в `src/main/java/` — общие для всех профилей
- Если фича требует compat-специфичного API — добавляй метод во **все** 4 `VersionCompat.java`
- CI собирает 4 JAR на каждый пуш в master автоматически
- Yarn-маппинги для old-профилей замораживаются на последних известных версиях — новых Yarn-релизов для них не будет, это нормально

**Правило для main-кода после migrateMappings:**  
После ренейма `src/main/java/` будет использовать Mojang-имена. Loom при сборке с `net.fabricmc.fabric-loom-remap` (старый plugin ID, используется для 1.21.x профилей) умеет работать с Mojang-именами в исходниках через intermediary, так что old-профили **не сломаются**. Если всё же возникнет конфликт — понадобится разделить затронутый файл через VersionCompat.
